# Wind farm offering simulation

This repository contains the code for testing different offering strategies for an offshore wind farm in the day-ahead and balancing markets. The setup is inspired by the Assignment 2 of the course "Renewables in Energy Markets" at DTU by Professor Jalal Kazempour in 2025. The code is structured in a modular way, allowing to easily add new data sources, scenario generators, and offering strategies.

## Getting Started

### Prerequisites
- Python 3.11+
- Git

### Installation

1. **Clone the repository**
```bash
   git clone git@github.com:dguiraldes/optimal_offer.git
   cd optimal_offer
```

2. **Create and activate a virtual environment**

   macOS/Linux:
```bash
   python -m venv .venv
   source .venv/bin/activate
```

   Windows:
```bash
   python -m venv .venv
   .venv\Scripts\activate
```

3. **Install the project and its dependencies**
```bash
   pip install -e .
```

That's it. The `src/` package is now importable from anywhere in the project,
including notebooks.


### Environment variables
Some data sources require API keys. Copy the example file and fill in your credentials:

Never commit the `.env` file — it is listed in `.gitignore`.

### Notebooks and data

Read through the notebooks in `notebooks/` and `model_training/` for data collection, data exploration, model training, scenario generation, strategy simulations and evaluations.

Data is not commited to the repository, but can be generated by running the notebooks.

## Data clients

All API clients live under `src/data/` as flat modules:

```
src/data/
    __init__.py          # Re-exports all API classes
    base.py              # BaseAPI with shared _get() and _resolve_params()
    energinet.py         # EnerginetAPI
    nordpool.py          # NordpoolAPI
    renewablesninja.py   # RenewablesNinjaAPI
```

Every client inherits from `BaseAPI`, which provides:
- `_get()` — HTTP GET with error handling
- `_resolve_params()` — resolves a raw `dict` or typed kwargs into query parameters

### Adding a new endpoint to an existing API

1. Open the corresponding module (e.g. `src/data/nordpool.py`).
2. Define a params dataclass for the new endpoint:

```python
@dataclass
class CapacityPriceParams:
    date: str
    delivery_area: str

    def to_query_params(self) -> dict:
        return {
            "date": self.date,
            "deliveryArea": self.delivery_area,
        }
```

3. Add a public method to the API class:

```python
def capacity_prices(
    self,
    params: dict | None = None,
    *,
    date: str | None = None,
    delivery_area: str | None = None,
) -> pd.DataFrame:
    query = self._resolve_params(params, CapacityPriceParams, date=date, delivery_area=delivery_area)
    data = self._get(f"{_BASE_URL}/CapacityPrices", params=query)
    return self._parse_capacity_prices(data)
```

4. Add a static `_parse_*` helper to convert the JSON response into a DataFrame.

### Adding a new API source

1. Create a new module `src/data/newapi.py`:

```python
from dataclasses import dataclass

import pandas as pd

from src.data.base import BaseAPI

# -- Param models --

@dataclass
class SomeEndpointParams:
    start: str
    end: str

    def to_query_params(self) -> dict:
        return {"start": self.start, "end": self.end}

# -- Client --

_BASE_URL = "https://api.example.com"

class NewAPI(BaseAPI):
    """Client for the New API."""

    def some_endpoint(
        self,
        params: dict | None = None,
        *,
        start: str | None = None,
        end: str | None = None,
    ) -> pd.DataFrame:
        query = self._resolve_params(params, SomeEndpointParams, start=start, end=end)
        data = self._get(f"{_BASE_URL}/some-endpoint", params=query)
        return pd.DataFrame(data["results"])
```

2. Register it in `src/data/__init__.py`:

```python
from src.data.newapi import NewAPI

__all__ = [
    ...,
    "NewAPI",
]
```

Pre-trained model weights are stored in `trained_models/` and training notebooks

### Using a generator

```python
from src.scenarios.wind_production import WindProductionGenerator

# Load with the default pre-trained checkpoint
gen = WindProductionGenerator(model="conditional_nvp")

# Or point to a specific checkpoint
gen = WindProductionGenerator(model="conditional_nvp", model_path="path/to/model.pt")

# Check required inputs
print(gen.required_inputs)  # ['wind_forecast']

# Generate 200 scenarios conditioned on a day-ahead forecast (numpy array, shape (96,))
scenarios = gen.generate(n_scenarios=200, wind_forecast=forecast)
# scenarios.shape == (200, 96)
```

All generators follow the same interface — swap `WindProductionGenerator` for
`SpotPriceGenerator`, `ImbalanceDirectionGenerator`, or `ImbalancePriceGenerator`.

```python
from src.scenarios.wind_production.my_model import MyModel
```

## Market module

The market module lives under `src/market/` and provides three layers: profit
evaluation, stochastic optimisation, and strategy benchmarking.

```
src/market/
    __init__.py             # Re-exports all public classes
    profit_evaluation.py    # OnePriceEvaluator, TwoPriceEvaluator
    model_MaxProfitCVar.py  # BaseModel, OnePriceModel, TwoPriceModel
    strategy_evaluation.py  # DayEvaluator, StrategyResults
```

### Settlement models

Two imbalance settlement schemes are supported, both in the evaluators and the
optimisation models.

| Class | Settlement rule |
|---|---|
| `OnePriceEvaluator` | Imbalance settled at a single imbalance price regardless of grid direction |
| `TwoPriceEvaluator` | Surplus/deficit priced asymmetrically depending on the grid's dominant direction |

All evaluators accept array inputs (1-D or 2-D with a scenario axis) and expose
three attributes: `da_profit`, `imb_profit`, and `total_profit`.

### Stochastic optimisation models

`OnePriceModel` and `TwoPriceModel` solve a mean–CVaR stochastic programme
to find the optimal day-ahead bid given a scenario cube:

```python
from src.market import OnePriceModel, TwoPriceModel

# scenarios shape: (4, n_scenarios, 96)
#   axis 0: [wind power (MW), spot price, imbalance direction, imbalance price]
model = OnePriceModel(scenarios, P_max=500)

# Pure expected-profit maximisation (no risk management)
model.fit(beta=0)

# Mean-CVaR (alpha=0.95, beta=0.75 → 75 % weight on CVaR)
model.fit(beta=0.75, alpha=0.95)

# Retrieve the optimal bid vector (shape: (96,))
p_DA_opt = model.p_DA_opt
```

The `beta` parameter interpolates between pure expected-profit maximisation
(`beta=0`) and pure CVaR maximisation (`beta=1`). `alpha` sets the confidence
level of the CVaR tail (e.g. `0.95` → worst 5 % of scenarios).

`TwoPriceModel` uses the same interface but internally introduces auxiliary
variables for surplus (`delta_plus`) and deficit (`delta_minus`) to handle the
asymmetric two-price settlement in the objective.

### Strategy evaluation

`DayEvaluator` evaluates any bid vector against both the scenario distribution
and realised outcomes for a single trading day:

```python
from src.market import DayEvaluator

evaluator = DayEvaluator(
    date="2026-02-10",
    imbalance_settlement="one_price",   # or "two_price"
    scenarios=in_sample_scenarios,      # shape (4, n_scenarios, 96)
    df_real=df_real_date,               # 96-row DataFrame with real observations
)

result = evaluator.evaluate_strategy(
    p_DA=p_da_bid,          # shape (96,)
    strategy_id="my_strat",
    strategy_name="My Strategy",
    alpha_CVaR=0.95,
)

print(result.profit_real)       # EUR, realised against observations
print(result.expected_profit)   # EUR, mean across scenarios
print(result.cvar_alpha)        # CVaR at the chosen confidence level
```

`evaluate_strategy` returns a `StrategyResults` dataclass with the full set of
performance metrics, including scenario-based metrics (expected profit, std dev,
VaR, CVaR, probability of loss, Sharpe-like ratio, regret, efficiency, DA/imbalance
revenue split, bid bias) and realised metrics (`profit_real`,
`exposure_to_imbalance`).

### Implemented offering strategies

The following strategies are benchmarked in `notebooks/04-offer_strategies_opis.ipynb`
and compared over the full test period (2026-02-01 to 2026-03-24) in
`notebooks/05-strategy_comparison.ipynb`:

| Strategy | Description |
|---|---|
| Day-ahead forecast | Bid the Energinet day-ahead wind forecast directly |
| Expected value | Bid the mean of the in-sample wind scenarios |
| Quantile offer | Bid a chosen quantile (e.g. 25 %, 40 %, 60 %, 75 %) of the wind scenarios |
| Mean-CVaR (one-price) | Solve `OnePriceModel` with varying `alpha` and `beta` |
| Mean-CVaR (two-price) | Solve `TwoPriceModel` with varying `alpha` and `beta`, evaluated under one-price settlement |
